# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**genai-trend-sync** - 自分専用AIトレンド・キュレーター

### プロダクトコンセプト
- **自動収集**: 主要AI企業（OpenAI、Anthropic、Google等）の公式情報を定期的に自動収集し、情報収集コストをゼロにする
- **LLMによる再構成**: 単なる要約ではなく、読みやすく・記憶に残る形式にLLMが記事を再構成する（コア価値）
- **ビジュアル強化**: 記事ごとに概念を表現するイメージ図を画像生成AIで自動作成し、視覚的に理解を助ける
- **Notionへの統合**: テキストと画像を組み合わせたリッチな記事として、自分専用ナレッジベース（Notion）に自動投稿する

### プロダクトビジョン
生成AIの進化は速く、主要プレイヤーのアップデートをタイムリーに把握することが難しくなっている。
本プロダクトは「読むのが楽しみになる、自分専用のAIトレンド・キュレーター」を目指す。
システムが自動で情報を収集し、LLMが読者の理解を助ける文体で再構成し、さらにイメージ図を添えてNotionへ届けることで、情報収集・整理・理解の負担をなくし、インサイトの取得と知識の定着に集中できる環境を実現する。

### 目的
- 主要AI企業のアップデートを週次で自動収集し、情報収集にかかる工数をゼロにする
- 無味乾燥な要約ではなく、読んで面白い・頭に残る文体と、理解を助けるビジュアルを両立させる
- Notionを個人のAIトレンド・ナレッジベースとして自動的に充実させる
- 社内のテクノロジー部門への情報展開を効率化する

## ターゲットユーザー

### プライマリーペルソナ: 自分自身（テクノロジー部門所属・30代）
- 生成AIの動向を業務上追う必要があるが、情報量が多く全てはフォローしきれない
- Notionをナレッジベースとして活用しており、情報がそこに集約されると使いやすい
- 箇条書きの機械的な要約より、文脈や技術的インパクトが伝わる読み物を好む
- 図やビジュアルがあると概念の理解・記憶の定着が進むタイプ
- 週1回程度、まとめて読むスタイルが合っている
- **1日の典型的なワークフロー**: 月曜朝にNotionを開き、週次でまとめられたAIトレンド記事を確認 → 社内共有に活用

### セカンダリーペルソナ: 社内テクノロジー部門のメンバー
- AIトレンドのインプットを自力でする時間が取れない
- 信頼できる一次情報ベースでまとめられた記事を共有してもらいたい
- テキストだけでなく、ビジュアルがあると短時間でも内容を把握しやすい
- Notionのリンクを送るだけで情報共有が完結すると嬉しい

## 成功指標（KPI）

### プライマリーKPI
- **Notion投稿数**: 週次実行で週あたり3件以上の記事が投稿されること（初月から）
- **画像生成成功率**: Notionに投稿される記事のうち、イメージ図が正常に生成・添付されている割合が90%以上
- **継続稼働率**: GitHub Actions の週次ジョブが4週間連続で正常完了すること（初月）
- **コンテンツ品質**: 生成記事（テキスト + 画像）を読んで「面白い・参考になった」と感じる割合が80%以上（自己評価）

### セカンダリーKPI
- **情報収集工数**: 手動でのトレンド収集・整理にかける時間が週30分以下になること
- **ナレッジ活用**: Notionに蓄積した記事を社内共有に月1回以上活用できること

## 機能要件

### コア機能（MVP）

#### 1. AIトレンド情報の自動収集

**ユーザーストーリー**:
テクノロジー担当者として、主要AI企業の最新情報を手動で探す手間をなくすために、公式ブログ等のRSSフィードから記事を自動収集する機能が欲しい

**受け入れ条件**:
- [ ] OpenAI、Anthropic、Google DeepMind の公式ブログRSSから記事を取得できる
- [ ] 直近7日以内に公開された記事のみを収集対象とする
- [ ] 取得した記事のタイトル・URL・公開日・本文（またはサマリー）を構造化データとして保持する
- [ ] ソースが取得できない場合（接続エラー等）は該当ソースをスキップしてログに記録し、他ソースの処理を継続する

**優先度**: P0（必須）

---

#### 2. LLMによる記事の再構成と画像生成プロンプトの作成

**ユーザーストーリー**:
テクノロジー担当者として、生のRSS情報をそのまま読むのではなく、技術的インパクトや背景が伝わる読み物として楽しむために、LLMが記事内容を再構成し、同時に理解を助ける画像生成プロンプトも作成する機能が欲しい

**受け入れ条件**:
- [ ] 収集した記事ごとにLLM APIへリクエストし、指定フォーマットに沿った再構成テキストを生成できる
- [ ] 出力フォーマットは、**①概要（1〜2文）**、**②技術的インパクト・注目ポイント**、**③関連する背景・文脈**、**④所感・示唆**の4セクションを含む
- [ ] 単なる箇条書き要約でなく、読み物として自然な文体で生成される
- [ ] 同一レスポンス内、または連続したリクエストで、記事のコア概念を視覚化するための**画像生成用プロンプト（英語、100〜300語程度）**を出力する
- [ ] LLM APIがエラーを返した場合は該当記事をスキップし、エラーをログに記録する
- [ ] 使用するLLMモデルは設定ファイルで切り替え可能にする（初期値: **Gemini 2.5 Pro**）

**優先度**: P0（必須）

---

#### 3. イメージ図の自動生成

**ユーザーストーリー**:
テクノロジー担当者として、記事の概念を視覚的に把握しやすくするために、画像生成AIが記事ごとにアイキャッチ／概念図を自動生成する機能が欲しい

**受け入れ条件**:
- [ ] ステップ2で生成された画像プロンプトを画像生成APIへ渡し、1記事につき1枚の画像を生成できる
- [ ] 使用する画像生成モデルは設定ファイルで切り替え可能にする（初期値: **Gemini 2.5 Flash Image（通称 Nano Banana）**）
- [ ] 生成画像は一時的に保存し、Notion投稿時にアップロード／埋め込みできる形式（PNG または JPEG）にする
- [ ] 画像生成がエラー／セーフティフィルタで失敗した場合は画像なしで記事投稿を継続し、失敗をログに記録する（他記事の処理は止めない）

**優先度**: P0（必須）

---

#### 4. Notionデータベースへのリッチな記事投稿

**ユーザーストーリー**:
テクノロジー担当者として、再構成した記事と生成画像を手動でコピーせずに、Notionのナレッジベースへ自動で蓄積するために、Notion APIを使った自動投稿機能が欲しい

**受け入れ条件**:
- [ ] 指定したNotionデータベースに1記事 = 1ページとして投稿できる
- [ ] 各ページには **タイトル・ソース企業名・元URL・公開日・実行日・再構成テキスト・生成画像** を含める
- [ ] 画像はNotionページ内に表示されるよう埋め込む（アイキャッチ配置を想定）
- [ ] 同一URLの記事が既にNotionに存在する場合は重複投稿しない（冪等性の確保）
- [ ] Notion APIキーは環境変数から取得する
- [ ] 画像生成に失敗した記事は、画像なしでテキストのみ投稿する
- [ ] `NOTION_DATABASE_ID` が未設定の場合、`NOTION_PARENT_PAGE_ID` で指定した親ページ配下に **データベースを自動生成** する
- [ ] 自動生成されるデータベース名は **`YYYY-MM-DD-GenAI-Trend-News`**（実行日の日付）形式とする
- [ ] 自動生成時は Title / Source / URL / PublishedAt / SyncedAt / HasImage の各プロパティを自動設定する
- [ ] 同名のデータベースが既に存在する場合は新規作成せず既存のデータベースを再利用する（冪等性の確保）
- [ ] `NOTION_DATABASE_ID` も `NOTION_PARENT_PAGE_ID` も未設定の場合は起動時にエラーで終了する

**優先度**: P0（必須）

---

#### 5. GitHub Actions による週次定期実行

**ユーザーストーリー**:
テクノロジー担当者として、毎週手動でスクリプトを実行する手間なしに情報を受け取るために、GitHub Actions で週次自動実行される仕組みが欲しい

**受け入れ条件**:
- [ ] 毎週月曜日 JST 9:00 に自動実行されるワークフローが設定されている
- [ ] ワークフロー実行ログから成功・失敗・スキップ件数（記事単位 / 画像単位）を確認できる
- [ ] APIキー等のシークレット（LLM API、画像生成API、Notion API、GitHub）はGitHub Secrets経由で注入する
- [ ] 手動トリガー（`workflow_dispatch`）でも実行できる

**優先度**: P0（必須）

---

#### 6. テストモード（実行パラメータの動的指定）

**ユーザーストーリー**:
開発者／運用者として、プロンプトやモデル変更の動作確認をAPIコストを抑えて行うために、1回の実行で処理する記事数や遡る日数を動的に指定できるテストモードが欲しい

**受け入れ条件**:
- [ ] 実行パラメータとして **最大記事数** と **遡る日数** を指定できる（例: 記事2件、過去3日）
- [ ] 指定方法は以下の両方をサポートする:
  - **CLI実行時**: 環境変数またはコマンドライン引数（例: `MAX_ARTICLES=2 LOOKBACK_DAYS=3 npm start` / `--max-articles 2 --lookback-days 3`）
  - **GitHub Actions 手動トリガー時**: `workflow_dispatch` の入力値として GUI から指定できる
- [ ] パラメータ未指定時は本番運用の既定値で動作する（既定値: 最大記事数 = 各ソース5件、遡る日数 = 7日）
- [ ] テストモードでの実行であることがログに明示される（例: `[TEST MODE] max_articles=2, lookback_days=3`）
- [ ] テストモードでもNotion投稿の冪等性は維持される（同一URLの重複投稿は発生しない）
- [ ] テストモード実行時に、Notion投稿先をテスト用データベースに切り替えるオプションがある（環境変数 `NOTION_DATABASE_ID_TEST` の利用）

**優先度**: P0（必須）

### 将来的な機能（Post-MVP）

#### 7. 収集ソースの動的管理

設定ファイルまたはNotion上でRSSソースを追加・削除できる機能

**優先度**: P1（重要）

#### 8. 出力フォーマット／画像スタイルのカスタマイズ

ユーザーがテキスト再構成のプロンプトや画像生成のスタイル（フラット／スケッチ／写実等）を設定でカスタマイズできる機能

**優先度**: P1（重要）

#### 9. Slack / メール通知

Notion投稿完了後にSlackチャンネルやメールへ要約を通知する機能

**優先度**: P2（できれば）

#### 10. 過去記事の再生成

プロンプトやモデルを改善した際に、既存のNotionページのテキスト・画像を再生成できる機能

**優先度**: P2（できれば）

## 非機能要件

### パフォーマンス
- 1回の週次実行（3ソース × 最大5記事 = 最大15件処理、各記事で画像1枚生成）が **15分以内** に完了すること
- LLM APIへのリクエストは1件あたり **30秒以内** にレスポンスを受け取ること（タイムアウト設定）
- 画像生成APIへのリクエストは1件あたり **60秒以内** にレスポンスを受け取ること（タイムアウト設定）

### 信頼性
- 個々の記事処理が失敗しても、他の記事の処理を継続できること（部分的な障害に耐性を持つ）
- 画像生成の失敗がテキスト記事の投稿を妨げないこと（画像なしで投稿継続）
- 実行ログにより、何件収集・生成・投稿したか（テキスト／画像別）が毎回確認できること
- Notion投稿の冪等性を保証し、同一記事の重複投稿が発生しないこと

### セキュリティ
- APIキー（LLM API、画像生成API、Notion API）は環境変数 / GitHub Secrets で管理し、コードやログに出力しない
- `package.json` の依存ライブラリは既知の脆弱性のないバージョンを使用する（`npm audit` でチェック）
- 生成画像の一時保存先はワークフロー実行内に限定し、実行完了後に自動で破棄する

### 保守性
- 収集ソース（RSSエンドポイント）、使用LLMモデル、使用画像生成モデルは設定ファイル（`config/sources.json` 等）で管理し、コード変更なしに切り替えられること
- TypeScript の strict モードを有効にしてコンパイルエラーゼロを維持する

### スケーラビリティ
- ソース数を10件程度まで拡張しても、実行時間が30分以内に収まる設計にする（並列処理で対応）

---

## スコープ外

明示的にスコープ外とする項目:
- Webダッシュボード・GUIの提供（すべてGitHub Actions + Notionで完結）
- リアルタイム収集・プッシュ通知（週次バッチに限定）
- SNS（Twitter/X等）や動画コンテンツの収集・解析
- 複数ユーザーへの配信・マルチテナント対応
- 収集した記事の感情分析・定量トレンド分析
- 画像の手動編集・差し替え機能（MVPでは生成画像をそのまま利用）
